# EthereumからAptosへのチートシート

違いと類似点についてさらに詳しく学ぶには[Aptos Learn](https://learn.aptoslabs.com/en/tutorials/ethereum-to-aptos-guide/cheat-sheet?workshop=eth-to-aptos)をご覧ください。

### 概要

| 機能                    | Ethereum                                                                                                                       | Aptos                                                                                                                                 |
| -------------------------- | ------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------- |
| **スマートコントラクト**        | Solidity, EVM                                                                                                                  | Move, MoveVM                                                                                                                          |
| **利点**               | 成熟度が高く、広く採用されている                                                                                                          | スケーラビリティ、低レイテンシー、予測可能な手数料                                                                                            |
| **トランザクション手数料**       | 変動的で高額になる可能性あり                                                                                                          | より低額で予測可能                                                                                                            |
| **アカウントアドレス**      | 160ビット                                                                                                                        | 256ビット                                                                                                                               |
| **アカウント構造**      | 単一フィールドの残高、nonceを使用                                                                                          | モジュールとリソース、シーケンス番号を使用                                                                                           |
| **データストレージ**           | パトリシアマークルツリー                                                                                                          | リソースとモジュールによるグローバルストレージ                                                                                             |
| **ストレージの考え方**        | コントラクトベースのストレージ                                                                                                         | コードとデータのアカウント中心の考え方                                                                                             |
| **コード例**           | [ERC-20](https://github.com/OpenZeppelin/openzeppelin-contracts/tree/master/contracts/token/ERC20)                             | [ファンジブルアセット](https://github.com/aptos-labs/aptos-core/blob/main/aptos-move/framework/aptos-framework/sources/fungible_asset.move) |
| **呼び出し元ID**              | `msg.sender`                                                                                                                   | `&signer`参照                                                                                                                   |
| **アップグレード可能性**         | プロキシパターン                                                                                                                 | モジュールの直接アップグレード                                                                                                                |
| **安全性とセキュリティ**      | リエントランシーなどの攻撃に対して脆弱                                                                                          | 一般的な脆弱性を軽減                                                                                                      |
| **ディスパッチタイプ**          | 動的ディスパッチ                                                                                                               | 静的ディスパッチ                                                                                                                       |
| **FTスタンダード**            | [ERC-20](https://docs.openzeppelin.com/contracts/4.x/erc20)                                                                    | [Coin](../smart-contracts/aptos-coin.mdx)（レガシー）と[ファンジブルアセット](../smart-contracts/fungible-asset.mdx)  |
| **NFTスタンダード**          | [ERC-721](https://docs.openzeppelin.com/contracts/4.x/erc721), [ERC-1155](https://docs.openzeppelin.com/contracts/4.x/erc1155) | [デジタルアセット](../smart-contracts/digital-asset.mdx)                                                                      |
| **ブロックチェーンとの対話** | [Ethers.jsライブラリ](https://docs.ethers.org/v6/)                                                                               | [Aptos TypeScript SDK](../sdks/ts-sdk.mdx)                                                                                |

### トークンスタンダードの詳細比較

|                        | Solidity                                                          | Move (Aptos)                                                                                                                                                    |
| ---------------------- | ----------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **トークン構造**    | 各トークンが独自のコントラクト                                   | すべてのトークンは単一の再利用可能なコントラクトを使用する型付き`Coin`または`FungibleAsset`                                                                             |
| **トークンスタンダード**     | ERC20などの標準に準拠する必要あり。実装は様々   | すべてのトークンに統一されたインターフェースと実装                                                                                                            |
| **残高ストレージ**    | マッピング構造を使用してコントラクト内に残高を保存            | **リソース指向の残高**: 残高はユーザーアカウントのリソースとして保存。リソースは任意に作成できず、トークン価値の整合性を保証 |
| **転送メカニズム** | 受信者の明示的な許可なしにトークンを転送可能 | 特定のケース（AptosCoinなど）を除き、一般的にトークンの転送には受信者の`signer`権限が必要                                                |

<br />

### EVMとMove VMの詳細比較

- **EVM**: 柔軟性と動的ディスパッチで知られ、幅広いスマートコントラクトの動作を可能にします。ただし、この柔軟性は並列実行とネットワーク操作の複雑さにつながる可能性があります。
- **Move VM**: VMとプログラミング言語の間のより統合されたアプローチで、安全性と効率性に重点を置いています。そのデータストレージモデルはより良い並列化を可能にし、静的ディスパッチ方式はセキュリティと予測可能性を向上させます。

<br />

|                                 | EVM (Ethereum Virtual Machine)                                         | Move VM (Move Virtual Machine)                                                                                     |
| ------------------------------- | ---------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------ |
| **データストレージ**                | データはスマートコントラクトのストレージスペースに保存                  | データはスマートコントラクト、ユーザーアカウント、オブジェクトにまたがって保存                                                 |
| **並列化**             | 共有ストレージスペースにより並列実行が制限                             | 柔軟な分割ストレージ設計により、より多くの並列実行が可能                                              |
| **VMと言語の統合** | EVMとスマートコントラクト言語（Solidityなど）は別レイヤー | VMレイヤーとMove言語の間のシームレスな統合、MoveでRustのネイティブ関数を実行可能 |
| **重要なネットワーク操作** | ネットワーク操作の実装が複雑で間接的   | バリデータセット管理などの重要な操作がMoveでネイティブに実装され、直接実行可能     |
| **関数呼び出し**            | 動的ディスパッチにより任意のスマートコントラクト呼び出しが可能            | 静的ディスパッチはセキュリティと予測可能な動作に重点                                          |
| **型安全性**                 | コントラクトの型が型安全性を提供                         | Moveのモジュール構造とジェネリクスが堅牢な型安全性を提供                                                      |
| **トランザクション安全性**          | トランザクションの順序付けと安全性にnonceを使用                       | トランザクションの順序付けと安全性にシーケンス番号を使用                                                         |
| **認証されたストレージ**       | スマートコントラクトストレージで対応                                      | Moveのリソースモデルを活用                                                                             |
| **オブジェクトアクセス性**        | オブジェクトはグローバルにアクセス不可。スマートコントラクトのスコープに制限    | オブジェクトのグローバルアクセス性を保証                                                                        |
