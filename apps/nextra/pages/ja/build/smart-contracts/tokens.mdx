# Aptos トークン標準

[Aptos デジタルアセット標準](digital-asset.mdx)は、Aptos上の正規のNon-fungible Token(NFT)を定義します。Aptosは、[Fungible Asset標準](fungible-asset.mdx)を通じて、デジタルアセット標準を代替性などの機能で拡張するために、コンポーザビリティを活用しています。このコンポーザビリティの概念は、これらの構造の基礎となるデータモデル：[Moveオブジェクト](objects.mdx)データモデルから来ています。

このドキュメントの残りの部分では、AptosのトークンスタンダードがEthereumやSolanaの標準とどのように比較されるかについて説明します。

## データモデル

トークンを理解するために、まず異なるブロックチェーン間でのデータモデルを比較してみましょう。

### Ethereum

Ethereumには2種類のアカウントがあります：

- イーサの残高を保存する外部所有アカウント
- スマートコントラクトを管理し、永続的な状態のためのストレージを持つコントラクトアカウント。このストレージは関連するコントラクトによってのみ変更可能です。

新しいNFTコレクションを作成するために、クリエイターはブロックチェーンに自身のコントラクトをデプロイする必要があり、そのストレージ内にコレクションとNFTのセットを作成します。

### Solana

データとコードが共存するEthereumやAptosとは異なり、Solanaではデータとプログラムを別々のアカウントに保存します。Solanaブロックチェーンには2種類のアカウントがあります：

- 実行可能アカウントはコントラクトコードのみを保存します
- 実行不可能アカウントは、実行可能アカウントに関連付けられ、所有されるデータを保存します。

新しいNFTコレクションを作成するために、クリエイターは既にデプロイされているプログラムを呼び出して、新しいコレクションとNFTのセットを作成します。

### Aptos

Aptosの[アカウント](../../network/blockchain/accounts.mdx)は、スマートコントラクトとデータの両方を保存します。Ethereumとは異なり、スマートコントラクトに関連するデータは、[アカウント](../../network/blockchain/accounts.mdx)内の[リソース](../../network/blockchain/resources.mdx)または[オブジェクト](objects.mdx)内の全アカウントの空間に分散されています。例えば、コレクションとそのコレクション内のNFTは、スマートコントラクトを定義する別のアドレスで、異なるアドレスの個別のオブジェクトに保存されます。スマートコントラクト開発者は、NFTとコレクションに関連するデータをスマートコントラクトと同じアドレスや他のオブジェクトに保存することもできます。

Aptosでは、NFTを作成する方法が2つあります：

- [ノーコード標準](https://github.com/aptos-foundation/AIPs/blob/main/aips/aip-22.md)により、クリエイターは新しいコントラクトをデプロイすることなく、コントラクトを呼び出して新しいコレクションとトークンを作成できます。
- カスタムNFTコントラクトにより、クリエイターはコレクションのすべての側面を管理できるオブジェクトモデルを拡張してNFTをカスタマイズできます。

Aptosは、Ethereumが提供するカスタマイズ性と、Solanaのような新しいコレクション作成の簡便さのバランスを取っています。

Ethereumと同様に、Aptosはアカウントが所有するすべてのNFTのセットを判断するためにインデックス作成が必要ですが、Solanaではその必要はありません。

## トークン標準の比較

Fungible Token (FT)は最初に[EIP-20](https://eips.ethereum.org/EIPS/eip-20)で導入され、Non-Fungible Token (NFT)は[EIP-721](https://eips.ethereum.org/EIPS/eip-721)で定義されました。後に、[EIP-1155](https://eips.ethereum.org/EIPS/eip-1155)がFTとNFT、さらにはSemi-Fungible Token (SFT)を1つの標準に統合しました。

Ethereumのトークン標準では、トークンのコレクションを区別するために、各トークンが独自の個別のコントラクトコードをデプロイする必要があります。Solanaのアカウントモデルは、1つの汎用プログラムが様々なデータを操作できるように、コードを再利用できる別のパターンを可能にします。新しいトークンを作成するには、トークンをミントできるアカウントと、それらを受け取ることができる複数のアカウントを作成できます。ミントアカウント自体がコントラクトアカウントの代わりにトークンタイプを一意に決定し、これらはすべて実行可能なアカウントにデプロイされた1つのコントラクトに引数として渡されます。

AptosのトークンスタンダードのコレクションはSolanaといくつかの類似点を共有しており、特にFT、NFT、SFTを共通のオンチェーンコードでカバーする方法が似ています。各新しいトークンに新しいスマートコントラクトをデプロイする代わりに、クリエイターは必要な引数を持つコントラクト内の関数を呼び出します。呼び出す関数に応じて、トークンコントラクトはトークンのミント/転送/バーン/...を行います。

### トークンの識別

Aptosはトークンをグローバルストレージ内の場所である`Address`または`ObjectId`で識別します。コレクションは、クリエイターのアドレスとコレクション名によって決定される場所に保存されます。

Ethereumでは、コントラクトはコントラクトをデプロイするアカウントによって決定されるアカウントにデプロイされます。NFTはその後、コントラクト内のデータテーブルへのインデックスとして保存されます。

Solanaでは、NFTデータはプログラムアカウントとは独立して、ミントアカウントの下に保存されます。

### トークンのメタデータ

Aptosトークンは、dappsがトークンと対話するために最も一般的に必要とされるデータを含む`Token`リソース内にメタデータを持っています。例えば：

- `name`: トークンの名前。コレクション内で一意である必要があります。
- `description`: トークンの説明。
- `uri`: トークンに関する詳細情報のためのオフチェーンへのURLポインタ。アセットは画像や動画などのメディア、またはJSONファイル内のより多くのメタデータである可能性があります。
- `collection`: コレクションのObjectIdへのポインタ。

追加のフィールドは、クリエイター定義のリソースまたは汎用的なキーバリューマップを定義する`PropertyMap`リソースに保存できます。

Ethereumでは、そのようなプロパティの一部のみがメソッドとして定義されています。例えば、ERC-20の`name()`、`symbol()`、`decimals()`、`totalSupply()`や、ERC-721のオプションのメタデータ拡張の`name()`と`symbol()`と`tokenURI()`、ERC-1155も同様にそのオプションのメタデータ拡張で`uri()`メソッドを持っています。トークンメタデータは標準化されていないため、dappsはケースバイケースで特別な処理を行う必要があります。

Solanaでは、Token Metadataプログラムも同様に、トークンに関連する多数のメタデータフィールドを定義するMetadata Accountを提供しており、Aptosの`TokenDataId`で定義されている`collection`も含まれています。ただし、SolanaはAptosとは異なり、アセットの可変性を提供していません。Aptosと同様に、Token Metadata v1.1.0はカスタマイズされたプロパティのための`attribute`コンテナを提供しています。
