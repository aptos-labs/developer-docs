# Aptos Glossary

## A

### Accumulator Root Hash

- **アキュムレータルートハッシュ**は、[Merkleアキュムレータ](https://eprint.iacr.org/2009/625.pdf)のルートハッシュです。

### Account

- Aptosブロックチェーンの**アカウント**は、任意の数の[Moveモジュール](#move-module)と[Moveリソース](#move-resources)を格納するコンテナです。
- 各アカウントの状態は、コードとデータの両方で構成されています。
- アカウントは[アカウントアドレス](#account-address)によって識別されます。

詳細は[アカウント](blockchain/accounts.mdx)をご覧ください。

### Account Address

- **アカウントアドレス**は、Aptosアカウントのアドレスです。
- アカウントアドレスは、Aptosネットワーク上の特定の宛先を指します。アドレスは、ブロックチェーン上で2つの当事者間で交換される特定の量の資産の宛先と送信元を指定します。
- Aptosアドレスは64文字の16進文字列(32バイト)です。多くの場合、これらの文字列の前に`0x`が付き、最初の16個のアドレスでは先頭の0が省略されます(例: `0x1`)。

詳細は[アカウント](blockchain/accounts.mdx)をご覧ください。

### API

- **アプリケーションプログラミングインターフェース(API)**は、外部アプリケーションを介してAptosブロックチェーンノードとクライアントネットワークとやり取りできるようにするプロトコルとツールのセットです。Aptosはノードと通信するためのREST APIを提供しています。
- 詳細は[ドキュメント](../build/apis.mdx)をご覧ください。

### APT

**Aptosトークン(APT)**は、ネットワークとトランザクション手数料の支払いに使用されるAptosブロックチェーンのネイティブトークンです。

### Aptos

**Aptos**は、誰もが使えるレイヤー1ブロックチェーンです。Moveプログラミング言語を使用し、2022年10月17日にメインネットを立ち上げ、web3のユーザー体験を再定義しました。Aptosブロックチェーンは、スピード、セキュリティ、スケーラビリティ、信頼性、使いやすさの向上と低トランザクションコストを通じて、より良いユーザー体験を創造することに専念しています。「Aptos」という言葉は、オーロン語で「人々」を意味します。Aptosブロックチェーンの詳細については、[Aptos公式ウェブサイト](https://aptosfoundation.org)をご覧ください。

### AptosBFT

- **AptosBFT**は、AptosプロトコルのBFTコンセンサスアルゴリズムです。
- AptosBFTはJolteonに基づいています。

### Aptos Blockchain

- **Aptosブロックチェーン**は、Aptosネットワーク(バリデータのネットワーク)上のバリデータによって合意された不変のトランザクションの台帳です。

### Aptos Name Service (ANS)

- **Aptos Name Service (ANS)**は、Aptosブロックチェーンの分散型ネーミングアドレスサービスです。Aptos名は、公開鍵の代わりに使用される人間が読める_.apt_ドメイン名です。例えば_love.apt_などです。
- このサービスでは、登録されたドメインに加えてサブドメイン名も登録できます。詳細は[Aptosnames.com](https://www.aptosnames.com/)をご覧ください。

### Aptos Core

**Aptos-core**は、Aptosネットワークソフトウェアのコードを含む[オープンソースリポジトリ](https://github.com/aptos-labs/aptos-core/)です。Aptos-coreには以下のソフトウェアが含まれています：

- 確認済みトランザクションの不変の台帳を生成・保存するAptosブロックチェーン自体
- トランザクションを検証し、それらをAptosブロックチェーンの不変の台帳に追加するコンセンサスアルゴリズムを実装する検証プロセス

### Aptos Ecosystem

- **Aptosエコシステム**は、Aptosブロックチェーンネットワークのさまざまなコンポーネントとそれらの相互作用を指します。Aptosエコシステムには、コミュニティ、[コミュニティ主導のプロジェクト](https://aptosfoundation.org/ecosystem/projects/all)、[イベント](https://aptosfoundation.org/events)が含まれます。

### Aptos Explorer

- **[Aptos Explorer](https://explorer.aptoslabs.com/)**は、アカウント情報、バリデータ、トランザクションなど、Aptosブロックチェーンの詳細を調べるためのインターフェースです。
- Aptos Explorerは、ユーザーがAptosウォレットやブロックチェーン上の他のツールでの作業を検証するのに役立ちます。

### Aptos Framework

**Aptos Framework**は、ブロックチェーンの更新とオンチェーンデータの構造のためのパブリックAPIを定義します。支払い、財務、オンチェーンガバナンスというAptosの機能の3つの重要な柱のビジネスロジックとアクセス制御を定義します。これはMoveプログラミング言語で書かれた一連のモジュールとして実装され、Moveバイトコードとしてチェーン上に保存されます。

### Aptos Node

**Aptosノード**は、Aptosブロックチェーンの状態を追跡するAptosネットワークのピアエンティティです。Aptosノードには、[バリデータ](#validator)と[フルノード](#fullnodes)の2種類があります。

### Aptos Protocol

- **Aptosプロトコル**は、Aptosネットワーク内でトランザクションがどのように送信、順序付け、実行、記録されるかの仕様です。

### AptosAccount

- **`AptosAccount`**は、シーケンス番号、残高、認証キーなど、アカウントに関連するすべての管理データを保持するMoveリソースです。
- **`AptosAccount`**は、すべてのアカウントが必ず持つ唯一のリソースです。

### AptosAccount module

- **AptosAccountモジュール**は、特定の`AptosAccount.T`リソースに保持される管理データを操作するためのコードを含むMoveモジュールです。
- シーケンス番号のチェックまたは増加、通貨の引き出しまたは預け入れ、ガス預金の抽出のためのコードがAptosAccountモジュールに含まれています。

### Aptos Devnet

- [devnet](#devnet)を参照してください。

## B

### Blocks

- Aptosでは、ブロックは同時にコミットされる[トランザクション](#transaction)のバッチです。
- ブロック番号は、ブロックチェーン文献における「ブロックの高さ」に相当します。
- トランザクションはブロックではなく台帳バージョンによって参照されます。

### BlockSTM

- **BlockSTM**は、Aptos Labsチームによって開発された最先端の動的並列実行エンジンです。
- 動的並列性により、開発者はトランザクションの書き込みセットを静的に宣言するという設計上の制約に直面することなく、柔軟にアプリケーションを作成できます。
- 業界の複数のブロックチェーンで採用されています。
- 詳細は[ブログ記事](https://medium.com/aptoslabs/block-stm-how-we-execute-over-160k-transactions-per-second-on-the-aptos-blockchain-3b003657e4ba)と[a16z cryptoでのプレゼンテーション](https://www.youtube.com/watch?v=2SE5tqPzhyw)をご覧ください。

### Byzantine (Validator)

- コンセンサスプロトコルの仕様に従わず、プロトコルの正しい実行を損なおうとする**バリデータ**。
- BFTアルゴリズムは伝統的に、アルゴリズムの投票権の最大3分の1までがビザンチンバリデータによって保持されることをサポートしています。

### Byzantine Fault Tolerance (BFT)

- **ビザンチン障害耐性**(BFT)は、一定のしきい値以下の故障した、または「[ビザンチン](https://en.wikipedia.org/wiki/Byzantine_fault)」バリデータが存在する場合でも、分散システムが安全性と活性の保証を提供する能力です。
- AptosブロックチェーンはJolteonに基づくコンセンサスプロトコルであるAptosBFTを使用します。
- BFTアルゴリズムは通常、集合的に$N$票を保持する複数のエンティティ(Aptosネットワークのシステム適用では「バリデータ」と呼ばれる)で動作します。
- $N$は、悪意のある可能性のある$f$票を保持するバリデータの数に耐えられるように選択されます。
- この構成では、$N$は通常$3f + 1$に設定されます。最大$f$票を保持するバリデータは、オフライン、悪意のある、遅いなどの故障が許可されます。[誠実な](#honest-validator)バリデータが$2f + 1$票を保持している限り、一貫した決定についてコンセンサスに達することができます。
- これは、投票権の最大3分の1がコンプロマイズされたバリデータまたは故障したバリデータによって保持されている場合でも、BFTコンセンサスプロトコルが正しく機能できることを意味します。

## C

### CLI

- **コマンドラインインターフェース**は、Aptosブロックチェーンでの開発、ノードの運用、問題のデバッグに使用されるAptos CLIを指します。詳細は[Aptos CLIページ](../build/cli.mdx)をご覧ください。

### Client

- **クライアント**は、ブロックチェーンから情報を受け取り、トランザクションを管理するソフトウェアです。クライアントはAptosノードを通じてブロックチェーンとやり取りします。

### Code Labs

- **コードラボとチュートリアル**は、非代替性トークン(NFT)の発行におけるAptos CLIの使用など、さまざまなワークフローを示し、ユーザーがプロセスの仕組みを理解し、関連する機能をコードで使用できるようにします。ユーザーはアカウントに必要な資金がある場合、devnet、testnet、mainnetネットワークで使用される同じコードラボとチュートリアルの手順に従うことができます。

### Consensus

- **コンセンサス**は、バリデータのコンポーネントです。
- コンセンサスコンポーネントは、実行されるトランザクションのブロック、その順序、実行結果について、すべてのバリデータ間の調整と合意を担当します。
- Aptosブロックチェーンは、これらの合意されたトランザクションとそれらの実行結果で形成されます。
- コンセンサスコンポーネントは、Aptosブロックチェーン上のすべてのバリデータ間のセキュリティ、信頼、合意の達成に責任を負います。

### Consensus Protocol

- **コンセンサスプロトコル**は、トランザクションを受け入れるか拒否するか、およびトランザクションの順序と実行結果について合意するために、nバリデータによって集合的に実行されます。
- [BFT](#byzantine-fault-tolerance-bft)を参照してください。

## D

### Dapps

- **分散アプリケーション(dapps)**は、Aptosブロックチェーン上で自律的に実行されるプログラムまたはデジタルアプリケーションです。この機能を実現するために一般的にスマートコントラクトが使用されます。

### Devnet

- **Aptosデブネット**は、一連のバリデータテストノードを使用して実行されるAptosネットワークの公開デプロイメントインスタンスです。
- デブネットは、新しいアイデアを試すために構築されたAptosネットワークのデモンストレーションです。
- デブネットはデジタル決済システムをシミュレートし、デブネット上のコインには_実際の価値はありません_。
- デブネットは、開発者が所定のプロトコルをテストする機会を与えられるネットワークです。メインネットとは独立して動作するという点でテストネットと似ていますが、毎週リセットされます。

## E

### Ed25519

- **Ed25519**は、サポートされているデジタル署名方式です。
- より具体的には、AptosネットワークはRFC 8032で定義されているEd25519曲線上のPureEdDSA方式を使用します。

### Epoch

- **エポック**は、バリデータセットの再構成と他の管理アクションの間の期間です。現在のAptosメインネットでは、2時間ごとです。

### Event

- **イベント**は、トランザクションの実行効果のユーザー向け表現です。
- トランザクションは、任意の数のイベントをリストとして発行するように設計できます。例えば、`Coin<AptosCoin>`の送金は、送信者アカウントに対して`WithdrawEvent`を、受信者アカウントに対して`DepositEvent`を発行します。
- Aptosプロトコルでは、イベントはトランザクションの成功した実行が特定の効果をもたらしたことの証拠を提供します。(上記の例では)`DepositEvent`により、受信者は自分のアカウントに支払いが受け取られたことを確認できます。
- イベントはブロックチェーンに永続化され、[クライアント](#client)からのクエリに答えるために使用されます。

### Execution

- Aptosブロックチェーンの**実行**は、トランザクションのブロックを管理するAptosノードコンポーネントです。実行コンポーネントは成功したトランザクションを保存します。

### Expiration Time

トランザクションは**有効期限**を過ぎると無効になります。以下を仮定します：

- $Time_C$はバリデータ間で合意された現在の時刻です($Time_C$はクライアントのローカル時刻ではありません)。
- $Time_E$はトランザクション$T_N$の有効期限です。
- $Time_C > Time_E$でトランザクション$T_N$がブロックチェーンに含まれていない場合、$T_N$は決してブロックチェーンに含まれないことが保証されます。

## F

### Faucet

- **フォーセット**は、devnetでAPTを発行するサービスです。testnetについては[mintページ](/network/faucet)をご覧ください。
- devnetとtestnetのAPTには実際の価値はなく、開発目的でのみ使用されます。
- フォーセットの使用については、[Faucet API](../build/apis/faucet-api.mdx)をご覧ください。

### Fullnodes

- **フルノード**は、ネットワーク上でデータが最新の状態で保存されることを保証するクライアントです。他のフルノードとバリデータノードからブロックチェーンの状態とトランザクションを複製します。

### Fungible Asset

- **代替性アセット**は、通貨、株式、ゲーム内リソースなど、価値を失うことなく他の同一のアセットと交換可能なアセットです。例えば、APTは代替性アセットです。なぜなら1 APTを別の1 APTと交換できるからです。
- Aptosブロックチェーンで代替性アセットを作成するには、[アセット標準規格](../build/smart-contracts/aptos-standards.mdx#asset-standards)に従ってください。
- フリーズとバーンの保証された強制の欠如やERC-20の承認のようなプログラム可能な送金などの高度な機能の不足といった`aptos_framework::coin`の短所に対処する次世代のCoin標準規格です。

### Fungible Token

- レガシーAptosトークン標準規格(aptos_token::token)では、**代替性トークン**は他の同一のトークン(つまり、同じ`TokenId`を共有するトークン)と交換可能なトークンです。これは、トークンが同じ`creator address`、`collection name`、`token name`、`property version`を持つことを意味します。
- Aptosデジタルアセット標準規格(aptos_token_objects::token)では、**代替性トークン**はデジタルアセットリソースを含むメタデータオブジェクトを持つ代替性アセットです。

### Fungible Unit

- **代替性ユニット**は、代替性アセットの個々のユニットです。これらのユニットは同一で、価値を失うことなく交換可能です。例えば、各[Octa](#octa)(APTの最小単位)は代替性ユニットです。

## G

### Gas

- **ガス**は、ブロックチェーンネットワーク上の計算とストレージに対する支払い方法です。Aptosネットワーク上のすべてのトランザクションには一定量のガスが必要です。
- トランザクションに必要なガスは、トランザクションのサイズ、トランザクションの実行の計算コスト、トランザクションによって作成される追加のグローバル状態の量(例：新しいアカウントが作成される場合)に依存します。
- ガスの目的は、バリデータの限られた計算リソースとストレージリソースに対する需要を調整し、サービス拒否(DoS)攻撃を防ぐことです。

詳細は[ガスとストレージ手数料](blockchain/gas-txn-fee.mdx)をご覧ください。

### Gas Unit Price

- 各トランザクションは、送信者がガス1単位あたりに支払う意思のある**ガス単価**を指定します。
- トランザクションに必要なガスの価格は、ネットワークの使用に対する現在の需要に依存します。
- ガス価格は[Octa](#octa)で表されます。

詳細は[ガスとストレージ手数料](blockchain/gas-txn-fee.mdx)をご覧ください。

## H

### Honest (Validator)

- **誠実**とは、バリデータがコンセンサスプロトコルを忠実に実行し、ビザンチンでないことを意味します。

### Jolteon

- **Jolteon**は、[BFT](#byzantine-fault-tolerance-bft)コンセンサスプロトコルの最近の提案です。
- AptosBFT、Aptosネットワークのコンセンサスアルゴリズムは、Jolteonに基づいています。
- 安全性に関する推論を簡素化し、以前のコンセンサスプロトコルのいくつかのパフォーマンス制限に対処します。特に、HotStuffと比較してレイテンシーを33%削減します。

## I

### Indexer

- **[Indexer](../build/indexer.mdx)**は、Aptosの生データを取得、処理し、効率的にデータベースに保存してAptosブロックチェーンの状態への高速なアクセスを提供するコンポーネントです。
- 高レベルでは、indexerはgRPCストリームからデータを取得し、生のブロックチェーンデータを変換するプロセッサを実行し、変換されたデータをGraphQLエンドポイントを介して提供します。

## L

### Leader

- **リーダー**は、コンセンサスプロトコルのためにトランザクションのブロックを提案するバリデータです。
- リーダーベースのプロトコルでは、ノードは進捗を得るためにリーダーに合意する必要があります。
- リーダーは、現在の[ラウンド番号](#round-number)を入力として取る関数によって選択されます。

## M

### Mainnet

- **メインネット**は、稼働中の完全に機能するブロックチェーンを指します。メインネットネットワークは完全にデプロイされ、送信者から受信者へのデジタル通貨の送金機能を実行します。

### Maximum Gas Amount

- トランザクションの**最大ガス量**は、送信者がトランザクションに対して支払う準備のある最大のガス単位量です。
- トランザクションは、使用されるガスが最大ガス量を超えない場合にのみ正常に実行できます。
- 請求されるガスは、このトランザクションの処理に必要なガス単位にガス価格を掛けたものに等しくなります。
- トランザクションの実行中にガスが不足した場合、または実行中にアカウントの残高が不足した場合、送信者は使用したガスに対して請求され、トランザクションは失敗します。

詳細は[ガスとストレージ手数料](blockchain/gas-txn-fee.mdx)をご覧ください。

### Mempool

- **メモリプール**は、バリデータのコンポーネントの1つです。送信されたが、まだ合意されておらず実行されていないトランザクションのインメモリバッファを保持します。メモリプールは他の[フルノード](#fullnodes)からトランザクションを受け取ります。
- バリデータのメモリプール内のトランザクションは、現在のノードのJSON-RPCサービスと他のAptosノードのメモリプールから追加されます。
- 現在のバリデータがリーダーである場合、そのコンセンサスコンポーネントはメモリプールからトランザクションを取得し、ブロックを形成するトランザクションの順序を提案します。その後、バリデータクォーラムがその提案に投票します。

### Merkle Trees

- **マークルツリー**は、データの整合性と更新の効率的な検証を可能にする認証されたデータ構造の一種です。
- Aptosネットワークは、ブロックチェーン全体を、時間の経過に伴うトランザクションと状態の履歴を記録する単一のデータ構造として扱います。
- [マークルツリー](https://en.wikipedia.org/wiki/Merkle_tree)の実装により、アプリケーションのブロックチェーンへのアクセスが簡素化されます。アプリケーションは以下のことができます：
  - 任意の時点のデータを読み取る。
  - 統一されたフレームワークを使用してデータの整合性を検証する。

### Merkle Accumulator

- **[マークルアキュムレータ](https://www.usenix.org/legacy/event/sec09/tech/full_papers/crosby.pdf)**は、Aptosブロックチェーンが台帳を保存するために使用する_追加のみ_のマークルツリーです。
- マークルアキュムレータは、トランザクションがチェーンに含まれていることの証明(「包含証明」)を提供できます。
- 文献では「履歴ツリー」とも呼ばれています。

### Module

- Moveプログラミング言語の**モジュール**は、アセットを作成、送金、または保存できるプログラムまたはライブラリのいずれかです。

### Move

- **Move**は、Aptosブロックチェーン上のすべてのトランザクションを実装する新しいプログラミング言語です。
- [Moveスクリプト](#move-script)と[Moveモジュール](#move-module)という2種類のコードがあります。
- Moveは、アクセス制御と希少性を重視する安全で安心なweb3向けプログラミング言語です。これはAptosブロックチェーンの構築に使用されるプログラミング言語です。詳細は[Move on Aptos](blockchain/move.mdx)をご覧ください。

### Move Bytecode

- Moveプログラムは**Moveバイトコード**にコンパイルされます。
- MoveバイトコードはMoveスクリプトとMoveモジュールを表現するために使用されます。

### Move Module

- **Moveモジュール**は、Aptosブロックチェーンのグローバル状態を更新するためのルールを定義します。
- Aptosプロトコルでは、Moveモジュールは**スマートコントラクト**です。
- ユーザーが送信する各トランザクションにはMoveスクリプトが含まれています。Moveスクリプトは、1つ以上のMoveモジュールの手続きを呼び出して、ルールに従ってブロックチェーンのグローバル状態を更新します。

### Move Resources

- **Moveリソース**には、Move**モジュール**で宣言された**手続き**に従ってアクセスできるデータが含まれています。
- Moveリソースは、コピー、再利用、または紛失することはできません。これにより、Moveプログラマーが誤ってまたは意図的にリソースを失うことを防ぎます。

### Move Script

- ユーザーが送信する各トランザクションには**Moveスクリプト**が含まれています。
- これらのトランザクション(Moveスクリプトとも呼ばれる)は、クライアントがバリデータに送信する操作を表します。
- ユーザーが送信する各トランザクションには**Moveスクリプト**が含まれています。
- これらのトランザクション(Moveスクリプトとも呼ばれる)は、クライアントがバリデータに送信する操作を表します。
- この操作は、ユーザーAからユーザーBへのコインの移動要求である場合もあれば、公開された[Moveモジュール](#move-module)(スマートコントラクト)との対話を含む場合もあります。
- Moveスクリプトは、モジュールの手続きを呼び出すことによって、Aptosブロックチェーンのグローバルストレージに公開されたリソースと対話する任意のプログラムです。トランザクションのロジックをエンコードします。
- 単一のMoveスクリプトで、複数の受信者に資金を送金したり、複数の異なるモジュールから手続きを呼び出したりすることができます。
- Moveスクリプトはグローバル状態に保存されず、他のMoveスクリプトから呼び出すこともできません。これは一回限りのプログラムです。

Moveスクリプトの使用例については、[Moveスクリプト](../build/smart-contracts/scripts/script-tutorial.mdx)をご覧ください。

### Move Virtual Machine (MVM)

- **Move仮想マシン**は、[Moveバイトコード](#move-bytecode)で書かれたMoveスクリプトを実行して実行結果を生成します。この結果はブロックチェーンの**状態**を更新するために使用されます。
- 仮想マシンは[バリデータ](#validator)の一部です。
- Move仮想マシン(MoveVM)は、トランザクションと現在のブロックチェーン台帳の状態を入力として処理し、変更セットまたはストレージデルタを出力として生成する各バリデータノードを処理します。

## N

### Node

- **ノード**は、Aptosブロックチェーンの状態を追跡するAptosネットワークのピアエンティティです。
- Aptosノードは論理コンポーネントで構成されています。[メモリプール](#mempool)、[コンセンサス](#consensus)、[Move仮想マシン](#move-virtual-machine-mvm)はノードコンポーネントの例です。

### Nonce

- **ノンス**は一度だけ使用される数値で、認証プロトコルと暗号化ハッシュ関数のために生成されるランダムまたは準ランダムな数値です。

## O

### Octa

- **オクタ**は[APT](#apt)の最小単位です。1 APT = 10<sup>8</sup> オクタです。

### Open-Source Community

- **オープンソースコミュニティ**は、オープンソースソフトウェアに取り組む開発者グループを指す用語です。このグロッサリーを読んでいるあなたは、Aptosプロジェクトの開発者コミュニティの一員です。

## P

### Proof

- **証明**は、ブロックチェーン内のデータの正確性を検証する方法です。
- Aptosブロックチェーンのすべての操作は、それが実際に正しく、データが省略されていないことを暗号的に検証できます。
- 例えば、ユーザーが特定の実行済みトランザクションの情報を照会する場合、返されたデータが正しいことを示す暗号的な証明が提供されます。

### Proof-of-Stake (PoS)

**プルーフオブステーク(PoS)**は、ブロックチェーントランザクションの一意性と正当性を確認するセキュリティメカニズムです。PoSコンセンサスメカニズムは、バリデータのネットワークによって支えられるAptosブロックチェーンで活用され、バリデータはシステムを更新しトランザクションを処理します。

## Q

### Quorum Store
- **クォーラムストア**は、バリデータセット内でトランザクション(バッチ単位)を配信するコンポーネントです。
- リーダーのボトルネックを解消することで、コンセンサスのスループットを大幅に改善します。
- データの配信とメタデータの順序付けを分離し、バリデータが非同期に並行してデータを配信できるようにします。
- 詳細は[ブログ記事](https://medium.com/aptoslabs/quorum-store-how-consensus-horizontally-scales-on-the-aptos-blockchain-988866f6d5b0)をご覧ください。

## R

### Randapp

- ランダム性を機能として使用する[dapp](#dapps)です。

### Resource Account

- **リソースアカウント**は、ユーザーが管理するアカウントとは独立してリソースを管理するために使用されます。例えば、開発者はモジュールの公開やコントラクトの管理などのために、リソースアカウントを使用することができます。

- コントラクト自体は初期化後に署名者を必要としません。リソースアカウントにより、モジュールが他のモジュールに署名者を提供し、モジュールに代わってトランザクションに署名する手段が得られます。

使用方法については[リソースアカウント](../build/smart-contracts/resource-accounts.mdx)をご覧ください。

### REST API Service

- **REST APIサービス**コンポーネントは、Aptosノードの外部インターフェースです。送信されたトランザクションやクエリなど、すべての着信クライアントリクエストは、まずRESTサービスを通過する必要があります。クライアントがストレージや他のシステムコンポーネントにアクセスするには、RESTサービスコンポーネントを経由する必要があります。これによりリクエストがフィルタリングされ、システムが保護されます。
- クライアントが新しいトランザクションを送信するたびに、RESTサービスはそれを[メモリプール](#mempool)に渡します。

### Round

- **ラウンド**は、トランザクションのブロックとその実行結果についてコンセンサスを達成することで構成されます。

### Round Number

- **ラウンド番号**は、コンセンサスプロトコルの[エポック](#epoch)中にリーダーを選出するために使用される共有カウンターです。

## S

### SDKs

- Aptosの**ソフトウェア開発キット(SDK)**は、開発者がAptos上でカスタムアプリを素早く作成できるようにするツールセットです。詳細は[Aptos SDKを使用する](../build/sdks.mdx)をご覧ください。

### Sequence Number

- アカウントの**シーケンス番号**は、そのアカウントからチェーン上で送信され、コミットされたトランザクションの数を示します。そのアカウントから送信されたトランザクションが実行またはアボートされ、ブロックチェーンに保存されるたびに増加します。
- トランザクションは、送信者アカウントの現在のシーケンス番号と一致する場合にのみ実行されます。これにより、同じ送信者からの複数のトランザクションのシーケンスが可能になり、リプレイ攻撃を防ぎます。
- アカウントAの現在のシーケンス番号がXの場合、アカウントAのトランザクションTは、Tのシーケンス番号がXの場合にのみ実行されます。
- これらのトランザクションは、そのアカウントの次のシーケンス番号になるまで(または期限切れになるまで)メモリプールに保持されます。
- トランザクションが適用されると、アカウントのシーケンス番号はX+1になります。アカウントは厳密に増加するシーケンス番号を持ちます。

### Sender

- _別名_: 送信者アドレス
- **送信者**はトランザクションを発信するアドレスです。トランザクションは送信者によって署名される必要がありますが、複数の署名者を持つことができます。

### Smart Contract

- **スマートコントラクト**は、契約条件を自動的かつ直接的に実行するコンピュータプログラムを指します。
- 関連する詳細については[Moveモジュール](#move-module)をご覧ください。

### State

- Aptosプロトコルにおける**状態**は、分散データベースのスナップショットです。
- トランザクションはデータベースを変更し、新しく更新された状態を生成します。

### State Root Hash

- **状態ルートハッシュ**は、特定のバージョンにおけるAptosブロックチェーンの状態のすべてのキーと値に対する[マークルハッシュ](https://en.wikipedia.org/wiki/Merkle_tree)です。

## T

### Table

- [**テーブル**](https://github.com/aptos-labs/aptos-core/blob/main/aptos-move/framework/aptos-stdlib/doc/table.md)はTableタイプを実装し、Aptosでは大規模なアカウント内の情報をキーバリューデータとして保存するために使用されます。

関連するAptosのソースファイルについては[`table.move`](https://github.com/aptos-labs/aptos-core/blob/main/aptos-move/framework/aptos-stdlib/sources/table.move)をご覧ください。

### Testnet

- **テストネット**は、まだ完全に機能していないがdevnetよりも安定しているAptosネットワークを指します。メインネットのテスト用の代替ネットワークとして使用されます。

### Tokens

- **トークン**は、ブロックチェーン上で発行されるデジタル価値の単位です。保有する資産や価値と交換することができます。トークンには、代替可能トークン(FT)、非代替可能トークン(NFT)、準代替可能トークン(SFT)の種類があります。

### Transaction

- 生の**トランザクション**には以下のフィールドが含まれます：
  - [送信者(アカウントアドレス)](#account-address)
  - [Moveスクリプト](#move-script)
  - [ガス価格](#gas-unit-price)
  - [最大ガス量](#maximum-gas-amount)
  - [シーケンス番号](#sequence-number)
  - [有効期限](#expiration-time)
- 署名済みトランザクションは、デジタル署名が付加された生のトランザクションです。
- 実行済みトランザクションは、Aptosブロックチェーンの状態を変更します。

## V

### Validator

- _別名_: バリデータ
- **バリデータ**は、Aptosブロックチェーン上で検証を行うAptosエコシステムのエンティティです。クライアントからリクエストを受け取り、コンセンサス、実行、ストレージを実行します。
- バリデータはブロックチェーン上のすべてのトランザクションの履歴を維持します。
- 内部的に、バリデータは現在の状態を保持し、トランザクションを実行し、次の状態を計算する必要があります。
- Aptosのバリデータはトランザクションの検証を担当します。

### Validator Nodes

- **バリデータノード**は、コンセンサス、特にAptosにおけるビザンチン障害耐性(BFT)コンセンサスプロトコルに参加する特別なクラスのフルノードです。バリデータは、Aptosブロックチェーンに追加されるトランザクションとその追加順序について合意します。

### Version

- **バージョン**は、[トランザクション](#transaction)ごとに順次増加する数値です。
- Aptosでは、トランザクションはグローバルに順序付けられ、すべてのトランザクションはバージョン(ブロックチェーン用語では「高さ」とも呼ばれる)を持ちます。
- トランザクションバージョン0は最初のトランザクション(ジェネシストランザクション)で、トランザクションバージョン100はブロックチェーンの101番目のトランザクションです。

## W

### Well-Formed Transaction

Aptosトランザクションは、以下の各条件が真である場合に**整形式**とされます：

- トランザクションが有効な署名を持っている。
- 送信者アドレスにアカウントが存在する。
- 公開鍵を含み、公開鍵のハッシュが送信者アカウントの認証キーと一致する。
- トランザクションのシーケンス番号が送信者アカウントのシーケンス番号と一致する。
- 送信者アカウントの残高が[最大ガス量](#maximum-gas-amount)より大きい。
- トランザクションの有効期限が切れていない。
